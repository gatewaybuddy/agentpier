## Critical

### C1: CORS Wildcard Allows Credential Theft via Cross-Origin Requests
**File:** `src/utils/response.py`, `infra/template.yaml`  
**Finding:** Every response includes `Access-Control-Allow-Origin: *` and the SAM template sets `AllowOrigin: "'*'"`. Combined with API key auth via headers, any malicious website can make authenticated requests if a user/agent has the key available in a browser context.  
**Impact:** If any browser-based agent or dashboard sends API keys, a malicious site can steal them or perform actions on behalf of the agent.  
**Remediation:**
- Replace `*` with explicit allowed origins (`https://agentpier.io` and staging equivalent).
- In the SAM template, set `AllowOrigin` to specific domains.
- Add `Access-Control-Allow-Credentials` handling if cookies are ever introduced.

## High

### H1: Pagination Cursor Injection — Arbitrary DynamoDB ExclusiveStartKey
**File:** `src/handlers/listings.py` → `search_listings()`  
**Finding:** The pagination cursor is a base64-encoded JSON object passed directly as `ExclusiveStartKey`. While there's a check that `GSI1PK` matches the category, the cursor can contain arbitrary `PK`, `SK`, `GSI2PK`, `GSI2SK` values. DynamoDB's `ExclusiveStartKey` tells DynamoDB where to resume scanning the index — an attacker can craft a cursor that starts enumeration from any point in the GSI1 index, potentially enumerating data in unintended categories by manipulating `PK`/`SK` values within the cursor.  
**Impact:** Data enumeration beyond intended query scope. While GSI1PK is validated, the other key attributes in the cursor are not, potentially allowing cross-partition reads on the index.  
**Remediation:**
- Sign the cursor with HMAC (using a server-side secret) so it cannot be tampered with.
- Alternatively, only accept opaque server-generated tokens and store cursor state server-side.

### H2: Account Deletion Scan Doesn't Paginate — Incomplete Cascade
**File:** `src/handlers/auth.py` → `delete_account()`  
**Finding:** The listing cleanup uses `table.scan(FilterExpression=...)` which returns at most 1MB of data per call. If a user has many listings (or the table is large), the scan will be incomplete and orphaned listing records will remain. Additionally, ABUSE# and TRUST# records are never deleted.  
**Impact:** Orphaned records persist after account deletion — privacy violation (GDPR/CCPA), and ABUSE records with `client_ip` remain indefinitely (90-day TTL, but still).  
**Remediation:**
- Paginate the scan using `LastEvaluatedKey` in a loop.
- Delete `ABUSE#{user_id}` and `TRUST#{user_id}` records during account deletion.
- Consider using GSI2 (`AGENT#{user_id}`) to find listings instead of a full table scan.

### H3: Rate Limit Write-Before-Check Race Condition
**File:** `src/utils/rate_limit.py` → `check_rate_limit()`  
**Finding:** The rate limit writes the request record *before* checking the count. This means: (1) every request, even rate-limited ones, writes to DynamoDB — an attacker can generate unlimited DynamoDB writes; (2) the count is always off-by-one (counts the current request).  
**Impact:** DynamoDB write amplification attack. An attacker can spray requests to generate unbounded writes to the rate limit table, increasing costs.  
**Remediation:**
- Read count first, then write only if under the limit.
- Use DynamoDB atomic counters or conditional writes.

### H4: No Authentication on GET /listings, GET /listings/{id}, GET /trust/{user_id}
**File:** `src/handlers/listings.py`, `src/handlers/trust.py`  
**Finding:** Search, individual listing retrieval, and trust profile endpoints require no authentication. While this may be intentional for a public marketplace, it means:
- Any actor can enumerate all listings by category.
- Any actor can look up trust profiles for any user_id.
- Combined with no rate limiting on these endpoints, mass scraping is trivial.  
**Impact:** Complete data scraping of all listings and trust profiles. Competitor intelligence gathering. User enumeration via trust endpoint (try user_ids, see which exist).  
**Remediation:**
- Add rate limiting to read endpoints (even without auth).
- Consider requiring API key for search/trust with higher rate limits.
- At minimum, add WAF rate limiting for these paths.

## Medium

### M1: Content Filter Evasion — Unicode/Homoglyph and Separator Bypass
**File:** `src/utils/content_filter.py`  
**Finding:** The regex patterns use `\b` word boundaries and literal ASCII characters. Easily bypassed by:
- **Unicode homoglyphs:** `сocaine` (Cyrillic 'с'), `hеroin` (Cyrillic 'е')
- **Zero-width characters:** `co​caine` (zero-width space U+200B)
- **Leetspeak:** `c0caine`, `h3roin`, `f3ntanyl`
- **Separator insertion:** `c.o.c.a.i.n.e`, `co caine`
- **HTML entities / URL encoding** in description fields
- **Mixed case with diacritics:** `cocaïne`

The prompt injection patterns are also basic — `"ignore previous instructions"` is caught but `"disregard what you were told before"` or `"[SYSTEM] new context:"` are not.  
**Remediation:**
- Normalize text before filtering: strip zero-width chars, normalize Unicode (NFKD), convert homoglyphs to ASCII.
- Add leetspeak variants to patterns or add a normalization step.
- Consider an LLM-based content classifier as a second pass for high-value content.

### M2: Agent Name Squatting — No Reservation or Verification
**File:** `src/handlers/auth.py` → `register()`  
**Finding:** Agent names are checked for uniqueness via GSI1 query, but: (1) the check and put_item are not atomic — race condition allows duplicate names; (2) no validation beyond length — names like `OpenAI-Official` or `AgentPier-Support` can be registered for impersonation.  
**Remediation:**
- Use DynamoDB conditional writes (`ConditionExpression`) on the GSI1PK to prevent duplicates atomically.
- Add a reserved names list for brand protection.
- Validate agent_name format (alphanumeric + hyphens, no spaces).

### M3: Listing ID Not Validated — Path Traversal in DynamoDB Keys
**File:** `src/handlers/listings.py` → `get_listing()`, `update_listing()`, `delete_listing()`  
**Finding:** The `listing_id` from path parameters is used directly in DynamoDB key construction (`LISTING#{listing_id}`) with no validation. While DynamoDB doesn't have path traversal in the filesystem sense, a crafted `id` like `../../USER#abc` won't work because it becomes `LISTING#../../USER#abc`. However, an empty or excessively long ID is accepted.  
**Remediation:**
- Validate listing_id format: must match `lst_[a-f0-9]{12}`.
- Reject IDs that don't match the expected pattern.

### M4: Scan in delete_account Is a Cost/Performance Bomb
**File:** `src/handlers/auth.py` → `delete_account()`  
**Finding:** Full table scan with filter expression. As the table grows, this becomes O(n) on the entire table for every account deletion. At scale, this is a denial-of-wallet attack vector — register accounts and delete them to trigger expensive scans.  
**Remediation:**
- Use GSI2 (`AGENT#{user_id}`) to find user's listings instead of scanning.
- This index already exists and is purpose-built for this query.

### M5: Error Messages Leak Valid Category/Type Sets
**File:** `src/handlers/listings.py`  
**Finding:** Error responses for invalid category/type include the full set of valid values: `f"Invalid type. Must be one of: {VALID_TYPES}"`. This aids reconnaissance.  
**Impact:** Low direct impact but violates principle of least information.  
**Remediation:**
- Return generic messages: "Invalid category" without listing all valid options.
- Document valid values in public API docs instead.

### M6: No Input Validation on `pricing`, `contact`, `location`, `availability` Fields
**File:** `src/handlers/listings.py` → `create_listing()`, `update_listing()`  
**Finding:** These fields accept arbitrary JSON objects/strings with no schema validation. An attacker can store arbitrary large nested objects, inject HTML/JS in string fields (if ever rendered), or store unexpected data types.  
**Impact:** Storage abuse, potential XSS if listing data is rendered in a frontend without escaping, unexpected data breaking consumers.  
**Remediation:**
- Define and enforce schemas for `pricing` (e.g., `{amount: number, currency: string}`), `contact`, `location`.
- Set max size limits on each field.
- Run content filter on `contact` and `location` string values too.

## Low

### L1: DynamoDB TTL Deletion Is Not Immediate
**File:** `src/utils/rate_limit.py`  
**Finding:** DynamoDB TTL deletions can be delayed up to 48 hours. Rate limit records and auth failure records may persist beyond their intended window, potentially causing false positives on rate limiting.  
**Impact:** Minor — could block legitimate users slightly longer than intended.

### L2: authenticate() Swallows All Exceptions
**File:** `src/utils/auth.py`  
**Finding:** The entire `authenticate()` function is wrapped in a bare `except Exception: return None`. This means DynamoDB outages, SDK errors, or permission issues silently fail as "unauthorized" instead of returning a 500.  
**Remediation:**
- Log exceptions before returning None.
- Consider distinguishing between "invalid key" and "service error".

### L3: Registration Rate Limit Keyed Only by IP
**File:** `src/utils/rate_limit.py`  
**Finding:** Rate limits use source IP only. Behind shared NAT/VPN, multiple legitimate users share limits. Conversely, an attacker with multiple IPs bypasses limits trivially.  
**Impact:** Limited effectiveness of rate limiting against determined attackers.

## Informational

### I1: No API Versioning
The API has no version prefix (e.g., `/v1/listings`). Breaking changes will require careful migration.

### I2: No Request ID / Correlation ID
Responses don't include a request ID for debugging or support correlation.

### I3: trust_score Initialized to 0.0 in Listing but Recalculated Dynamically
In `create_listing()`, `trust_score` is set to `Decimal("0.0")` on the listing item, but `get_trust()` calculates it dynamically. The listing's embedded trust_score is never updated, so it's always stale.

### I4: No WAF Configuration in Template
The security doc mentions AWS WAF (100 req/5min/IP) but the SAM template doesn't define any WAF resources. This is presumably configured separately but should be documented or codified.

## Positive Observations

- **API key hashing:** SHA-256 hash stored, raw key shown once — good practice.
- **Key rotation:** Properly deletes old key records before creating new ones.
- **Ownership checks:** Update/delete listings verify `posted_by` — solid authz.
- **Content filter logging:** Violations are logged with TTL for review — good audit trail.
- **Auth failure tracking:** IP-based lockout after 5 failures — good brute force protection.
- **No hardcoded secrets:** No credentials, keys, or tokens found in source code.
- **Least privilege IAM:** Read-only functions get `DynamoDBReadPolicy`, write functions get `DynamoDBCrudPolicy`.
